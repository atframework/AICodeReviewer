# M10 — 模型元数据 Catalog（基本完成）

## 范围

- 使用 models.dev 元数据补齐 `ModelSpec`。
- 结构化刷新缓存、打包保底快照与离线回退。
- Gateway 成本估算、runtime bundle manifest 与 agent adapter 元数据注入。

## 主要交付

- Config schema 已支持 `llm.model_catalog`、provider `catalog_provider` / `catalog_id`、overrides 与 cache backend 校验。
- Store schema 已新增 `model_catalog` / `model_catalog_source` keyed 表与迁移 `003_model_catalog`。
- `packages/llm/src/model-catalog.ts` 负责 models.dev `api.json` 的纯解析、归一化和 `ModelSpec` 字段映射。
- `packages/server/src/model-catalog-service.ts` 提供刷新周期、ETag、三层回退、打包保底 seed、provider/model 匹配、用户覆盖合并，以及 Redis 结构化缓存 backend。
- `bootstrapServerApp` 已在 gateway 创建前初始化 catalog service 并充实 primary / fallback / summarize `ModelSpec`；store 初始化条件覆盖 admin auth、catalog SQLite backend 和 reflection memory。
- Gateway 成本估算使用 catalog 的真实价格字段，不再只依赖固定占位估算。
- Kilo / Roo / opencode / Claude Code / Copilot CLI 的 runtime metadata 注入或降级状态已写入 manifest。
- 打包保底快照 `packages/llm/assets/model-catalog/models-dev.json` 与刷新脚本已纳入仓库。
- 示例配置、示例 README、架构文档与决策索引已同步 M10 合同。

## 验收沉淀

- Config、store、llm、server、agents 相关单元测试覆盖解析、回退、bootstrap 顺序、gateway 价格和 adapter 注入矩阵。
- Redis backend 字段现在会先过配置交叉校验；bootstrap 阶段要求 `storage.cache.kind: redis` 且 `redis.url_env` 可解析，随后使用 `ioredis` 加载/持久化结构化 catalog cache。
- 用户显式配置永远优先于 catalog；catalog 只填补缺失字段。

## 当前稳定落点

- 详细合同：`../architecture.md` §3.13。
- 配置真源：`packages/core/src/config.ts`。
- Store 真源：`packages/store/src/schema.ts`、`packages/store/src/model-catalog.ts`。

## 遗留项

- 真实外部 Redis smoke/e2e：本地合同层已覆盖 Redis backend get/upsert/source metadata、model id 查询、bootstrap enrichment 与缺失配置显式拒绝；若需要验证具体部署环境的 Redis 版本、网络 ACL 或 TLS 参数，应作为外部系统验收项单独执行。
