# AICR Known Codebase Pitfalls

This is the on-demand companion for the root `AGENTS.md`. Read it before non-trivial implementation, review, roadmap, or AI-asset maintenance work, especially when touching config/schema, store migrations, agent runtime, VCS context, output channels, deployment, docs site, or workflow behavior.

Keep this file concise and actionable: each entry should name the fragile contract, the file or surface to check, and the mistake to avoid. Do not add chronological incident logs.

These issues have been found and fixed in prior sessions. Before making changes, check that you are not reintroducing any of them:

1. **Config schema must track Plan.md §3.10 / `docs/ai/architecture.md` §3.10**: `packages/core/src/config.ts` Zod schemas must include `compression`, `llm.fallback_chain`, `llm.retry`, `llm.budget`, `llm.per_provider_overrides`, `llm.model_catalog` (with `cache.backend` cross-field superRefine requiring `storage.cache.kind: redis` + `redis.url_env`), `llm.providers[].catalog_provider`/`catalog_id`, `queue.workers`, `queue.rate_limit`, `queue.retry`, `queue.dead_letter`, `queue.sqlite`, `review.problem_issue.max_recent_issues`, `review.reflection.memory`, `workspaces.defaults.agent`, `outputs.channels[].mention_fallback`, `outputs.routes`, `storage` (database/cache/object/retention), and `admin` (username_env/password_env/password_hash_env/session_ttl). If you add config fields, add corresponding tests in `packages/core/test/config.test.ts`.
2. **Store schema must track Plan.md §3.11 / `docs/ai/architecture.md` §3.11**: `packages/store/src/schema.ts` must include `triggerName`, `provider`, `providerModel`, and the full observability schema: `projects` (with `deleted_at`), `reviewRuns` (with `projectId` FK, `skipReason`, `compressed`, token estimates, target metadata, duration, problem/summary/dispatch counts), `codeMetrics`, `llmUsage`, `outputEvents`, `dailyRollups`, and the M10 model-catalog tables `modelCatalog` (keyed `catalog_id` PK + `provider_id`/`model_id` columns for fuzzy match + `data` JSON + `source` + `fetched_at`) and `modelCatalogSource` (source-level refresh metadata). New tables need a corresponding inline migration in `packages/store/src/database.ts` `MIGRATIONS` and `packages/store/test/schema.test.ts`.
3. **`isPlainObject` must reject built-in class instances**: `Date`, `RegExp`, and other non-plain prototypes must return `false`. Only `Object.prototype` or `null` prototype are plain.
4. **`normalizePath` must compress consecutive slashes**: `//` → `/` in addition to backslash replacement and leading `./` stripping.
5. **`estimateTokens` must handle CJK characters**: Characters in CJK Unicode ranges count as ~2 tokens, not 0.25.
6. **Prompt manager must detect conflicts**: `discoverRepoPromptAssets` must detect duplicate skill names, overlapping path-instructions, and alias vs copilot-instruction overlaps.
7. **Review orchestrator status logic**: When `dryRun` is false, status must not fall back to `"dry_run"` regardless of whether an output publisher exists.
8. **No unused imports**: Always verify that every import is used before committing. The CLI `app.ts` previously had an unused `ReviewEvent` import.
9. **DRY utility functions**: `normalizePath`, `normalizeChangedPath`, and `isPlainObject` live in `packages/core/src/utils.ts`. Import from `@aicr/core` or the local `./utils.js` — do not duplicate these functions in other files.
10. **Sandbox and agents packages must have tests**: Even if they only export a constant, create `test/index.test.ts` to verify the export.
11. **Sandbox `exactOptionalPropertyTypes`**: When passing optional fields from config to sandbox/agent constructors, use conditional spread (`options.image ? { image: options.image } : {}`) instead of direct assignment to avoid `undefined` type mismatches.
12. **Agent adapter factory must guard optional fields**: `createAgentAdapter` must not pass `undefined` binary to `createKiloAdapter`; use conditional construction instead.
13. **`execFile` does not support `input` on Node 20**: For passing stdin to child processes in sandbox/docker, use `spawn` directly rather than `execFile` with `input` option.
14. **Serve bootstrap must publish when configured**: `bootstrapServerApp` must not force `dryRun: true`; use a per-event `outputPublisherResolver` so Gitea webhook payloads can resolve PR numbers and publish line comments.
15. **Container sandbox engine and allowlist must be enforced**: Docker-compatible sandbox code must invoke the resolved `docker`/`podman` CLI and validate commands against `ALLOWED_COMMANDS` before spawning containers.
16. **Container env files must stay outside mounted workspaces**: `--env-file` paths for docker/podman must be temporary host files outside `agent/`, `tmp/`, and `source/` mounts, then deleted after the run.
17. **`fixListMarkerSpacing` must not break `**bold**` or `***bold-italic***`**: The `LIST_MARKER_WITHOUT_SPACE_RE` regex in `packages/core/src/markdown-fixer.ts` must exclude `*` from the follow-character set for `*` markers so that `**Text**` at line start is never split into `* *Text**`.
18. **`fixListMarkerSpacing` must not break thematic breaks**: Its indentation groups must use `[ \t]*`, not `\s*`, so `---` stays a thematic break instead of becoming `- --`.
19. **Keep generic public modules platform-neutral**: Shared/public modules such as `packages/cli/src`, `ReviewEvent`, and `TemplateContext` must not duplicate provider/channel literal lists or expose platform-specific field names. Import canonical schemas/constants from `@aicr/core` and use generic fields such as `sourcePath` and `submitterWorkspace`; keep Gitea/GitHub/P4/etc. names inside config contracts, docs/examples, tests, and platform-specific adapters.
20. **`fixAndValidateMarkdown` must enforce MD022 blanks-around-headings**: `fixBlanksAroundHeadings` in `packages/core/src/markdown-fixer.ts` must insert blank lines before and after ATX headings (`#`-prefixed lines) while skipping fenced code block content. This ensures LLM-generated summaries pass markdownlint MD022.
21. **Runtime bundle must be materialized for every agent run**: `materializeRuntimeBundle` in `packages/agents/src/runtime-bundle.ts` must be used instead of calling `adapter.materializeConfig` directly. It writes instructions, skills, MCP tool metadata, and a `manifest.json` into the isolated `agent/` directory.
22. **`renderMarkdownCodeFence` argument order**: The signature is `(content, language?)` — code snippet first, language second. All call sites must pass `(codeSnippet, codeLanguage)`, not `(codeLanguage, codeSnippet)`. The consolidated issue body builder in `packages/outputs/src/index.ts` previously had these swapped, producing garbled language identifiers in place of fenced code.
23. **IM markdown table regex must not use `g` flag with `.test()`**: `TABLE_ROW_RE` and `TABLE_DIVIDER_RE` in `packages/outputs/src/im-markdown.ts` must NOT have the `g` (global) flag. With `g`, `.test()` advances `lastIndex` between calls, causing consecutive table rows to be misdetected. Use `mu` flags only.
24. **PR review dispatchers must use `toWeComMarkdown`/`toFeishuMarkdown` for IM channels**: IM bot dispatchers (`createWeComBotDispatcher`, `createFeishuBotDispatcher`) must pass content through their respective `toXxxMarkdown()` transformer before sending. Raw `sections.join("\n")` breaks tables and headings in IM-rendered cards.
25. **PR review `review_mode` and `review_event` must flow from config**: `packages/server/src/bootstrap.ts` must read `review_mode` and `review_event` from channel config and pass them to `createGiteaPullRequestReviewDispatcher` / `createGithubPullRequestReviewDispatcher`. Default is `review_mode: "auto"` (try review API, fallback to issue comment on 403/422) and `review_event: "COMMENT"`.
26. **Agent free-form stdout is not a final report**: Agent-adapter runs must not publish natural-language stdout as an IM summary. Disable natural-language fallback for agent stdout, trigger a structured repair pass, and require `aicr.report_problem` records for any issue locations shown in Feishu/WeCom reports.
27. **Summary-only issue claims and missing-context skips must be repaired**: If output text says issues were found but `problemCount` is still zero, or if skip/summary prose asks humans to provide diff/source context, do not rely on `no_problems` routing. Clear provisional outputs and require structured `aicr.report_problem` records or `aicr.fetch_more_context`; for P4 context gaps, the adapter can `p4 print` full changed or related files at the reviewed changelist.
28. **Agent repair must fall back to direct LLM when model cannot produce structured output**: Models like `glm-5` may output natural-language prose instead of structured JSON tool calls when run through agent adapters (Kilo). When the agent repair retry still produces no parseable output, the orchestrator must bypass the agent and call `options.llm.complete()` directly with the repair prompt and `allowNaturalLanguageSummary: true`; however, prose that explicitly says there are no actionable problems or no reviewable code must be normalized to `aicr.skip` instead of publishing the generic format-repair fallback summary. Additionally, `extractKiloJsonStreamContent` must capture `tool_call`/`tool_use` stream events so that MCP tool calls made by the agent inside the sandbox are not silently dropped.
29. **Kilo MCP tool name format and MCP state file**: Kilo (≥7.x) prefixes MCP tool names with the server name and converts dots to underscores (e.g., `aicr.report_problem` → `aicr-output_aicr_report_problem`). `normalizeToolName` must map this format back to the canonical `aicr.*` names. The MCP output server writes `.aicr-output-state.json` to the agent workspace after each tool call; the orchestrator must read this file after the agent run and populate the AICR output collector from it, rather than relying solely on text-based JSON parsing.
30. **PR review update mode must use issue comment API for PATCH**: When `review_update_strategy: "update_existing"` is set, the summary comment is managed via issue comment endpoints (`GET` to list, `PATCH` to update, `POST` to create). The `<!-- aicr:managed=pr-review -->` marker must be present in every managed comment body. The `<!-- aicr:problems=... -->` marker tracks current fingerprints and must be regenerated on each update. Resolved fingerprints must appear in the body's Resolved section with their original value, not silently dropped.
31. **Review deduplication key must isolate trigger/workspace and use stable targets**: `buildDedupKey` in `packages/server/src/review-deduplicator.ts` must include `triggerName` and `workspaceId`, then prioritize `reviewEvent.branch` over `url`, `headSha`, or `baseSha` so that repeated `/aicr review` commands on the same PR/MR are deduplicated across new commits without collapsing unrelated comment-triggered reviews into one `unknown` target.
32. **Config/workflow/public changes must sync docs in the same change**: When code changes affect config shape, agent adapters, MCP tool contracts, output rendering, deployment behavior, public workflow, or review orchestration semantics, update the matching `Plan.md` roadmap summary, relevant `docs/` modules, `example/config.yaml`, and `example/README.md` entries in the same change, or explicitly state why no doc/example update is needed. This rule exists because prior sessions repeatedly missed doc sync, leading to stale contracts.
33. **Rootless Podman deploy scripts must specify `--storage-driver=overlay`**: When the server uses a custom `rootless_storage_path` in `/etc/containers/storage.conf`, Podman 5.x rootless auto-detection can fail after reboot or OOM, producing the misleading error `"invalid internal status ... could not find any running process"`. The real cause is storage-layer initialization failure, not a missing pause process. `podman system migrate` may crash with a nil pointer in this state. Always use `podman --storage-driver=overlay system migrate` to recover, and add `--storage-driver=overlay` to all `podman build/run/rm` commands in `deploy.sh`. Add a pre-flight check (`podman ps || podman --storage-driver=overlay system migrate`) at the top of deploy scripts.
34. **Native MCP context requests must trigger VCS follow-up**: Kilo MCP state `contextRequests` and JSON stream `tool_call` / `tool_use` events for `aicr.fetch_more_context` must be replayed through the orchestrator's VCS-backed handler and followed by a final pass. Clear `.aicr-output-state.json` before each agent run so stale context or summary state cannot publish "无法访问完整仓库代码" as a final report.
35. **Container sandbox workdir must be `/workspace/agent`**: Docker/Podman sandbox runs must set the container workdir to the writable agent mount. Otherwise Kilo-spawned MCP servers can write `.aicr-output-state.json` under the image workdir (for example `/app`) and the orchestrator will miss structured results.
36. **Runtime image copies need build-stage directories for hoisted-only packages**: `deploy/Dockerfile` must copy `packages/sandbox/node_modules` and `packages/eval/node_modules` alongside their `dist/` outputs, but `pnpm install` can omit those package-local directories when the packages only use hoisted/workspace dependencies. Create them in the build stage with `mkdir -p packages/sandbox/node_modules packages/eval/node_modules` before the runtime `COPY --from=build ... node_modules` lines, or remote image builds will fail with `copier: stat: "/app/packages/.../node_modules": no such file or directory`.
37. **pnpm 10.x native module builds require `onlyBuiltDependencies` in workspace yaml**: pnpm 10 introduced a build-approval gate for native modules. The correct fix is adding `onlyBuiltDependencies: [better-sqlite3]` to `pnpm-workspace.yaml`. Do NOT use `pnpm config set onlyBuiltDependencies` (sets a global string, not a project list), `--allow-build` (does not exist in pnpm 10.20.0), or `pnpm rebuild` (also blocked by the approval mechanism).
38. **P4-enabled runtime image requires Ubuntu/glibc**: `deploy/Dockerfile`
  now uses `ubuntu:24.04` as the distro base and copies the Node 22
  userspace from `node:22-bookworm-slim`. Official Perforce packages provide
  an Ubuntu APT repository and do not support Alpine/non-glibc images. Do
  not switch the runtime `BASE_IMAGE` back to Alpine/Wolfi/Chainguard unless
  you also replace the `p4-cli` installation path and revalidate the full
  tool baseline (`git`, `subversion`, `p4`, `ripgrep`, `fd`, `bat`, `jq`,
  `yq`, `kubectl`, `helm`, `podman`, `buildah`, `skopeo`, `python3-pip`,
  `cmake`, `ninja`, `clang-tidy`, `cppcheck`, `gdb`, `valgrind`,
  `shellcheck`, `strace`, `lsof`, `universal-ctags`).
39. **Windows PowerShell file encoding defaults to UTF-16 LE**: PowerShell 5.1 `>` redirect and `Out-File` default to UTF-16 LE encoding; `Out-File -Encoding utf8` adds a BOM. For remote file transfers (`.env`, `config.yaml`, scripts), use `Out-File -Encoding ascii` or `Set-Content -Encoding utf8` (PS 7+). Better: use `scp` from the Linux-side or `printf` on the remote host to avoid encoding issues entirely.
40. **Admin config uses `session_ttl_seconds` not `minutes`**: The `adminAuthSchema` in `packages/core/src/config.ts` uses `session_ttl_seconds` (default 28800 = 8 hours). Do not set `session_ttl_minutes`; it will be silently ignored and sessions will expire at the default.
41. **Config-only changes need container restart, not rebuild**: `config.yaml` and `.env` are volume-mounted into the container, not baked into the image. After editing either file, restart the container (`podman restart aicr`); a full image rebuild is only needed for code changes.
42. **PR review problems are buffered and flushed in `publishSummary`**: GitHub and Gitea PR review dispatchers buffer `publishProblem` calls and flush them as one consolidated Markdown reply body when `publishSummary` is called. Do not add tests that expect individual HTTP POSTs per problem or per-problem inline comments. The composite publisher must call summary flush for PR review channels that are only configured under `line_comments`, or buffered problems will be dropped.
43. **Resolved issues in PR review summary need readable labels**: The `<!-- aicr:problem-meta=BASE64_JSON -->` hidden marker in the PR review summary comment stores per-problem metadata (severity, category, file, line). Use it to render resolved items as readable titles with anchors. Legacy comments without metadata must show a readable placeholder such as `Previously reported issue`, never a raw fingerprint hash. If modifying `buildReviewSummaryCommentBody`, always include `buildReviewSummaryProblemMetaMarker`.
44. **Trigger retry prefers `queue.retry.attempts`**: The server reads `config.queue.retry.attempts` (default 1 = no retry) and `config.queue.retry.backoff` for trigger-level retry. Legacy `queue.retry.max_attempts` and `queue.retry.backoff_seconds` are normalized for compatibility, but new examples should use `attempts` and `backoff`. This is separate from `config.llm.retry.max_attempts`. The `onCompleted` dedup callback runs only once after the final attempt succeeds or all retries are exhausted.
45. **Reflection summary fingerprint must stay stable per workspace+pattern**: In `packages/core/src/reflection-extractor.ts`, the "Latest review summary" reflection must use a stable discriminator such as `stableFingerprint(workspaceId, "summary", "latest")`, never `runId`/`headSha`. A per-run fingerprint makes `writeReflectionMemory` insert a new row every commit instead of overwriting, so the most-recent-50 memory hints fill with duplicate summaries and crowd out the category/scope reflections. `runId` is still passed for `sourceRunId` provenance and reserved for future thorough-mode cross-run analysis, but it must not drive any fingerprint.
46. **Shared webhook extraction must preserve per-provider PR-detail fetchers**: When refactoring provider webhook translators into `packages/server/src/webhook-common.ts`, every provider that enriches `/aicr review` issue-comment events must keep passing its own `PullRequestDetailsFetcher` to `translateIssueCommentReviewCommand`. GitHub uses `Bearer` + `application/vnd.github+json`; Gitea/Forgejo use `Authorization: token <token>` + `application/json` (consistent with `issue-triage.ts`). Dropping the Gitea fetcher silently strips `headSha`/`baseSha` enrichment from comment-triggered Gitea PR reviews even though `buildWebhookConfigFromTrigger` populates `config.token`.
47. **Buffered PR-review problems must flush at most once per review**: The Gitea/GitHub PR review dispatchers in `packages/outputs/src/index.ts` buffer `publishProblem` and flush in `publishSummary`. Because the orchestrator can emit multiple summary entries, guard the non-`update_existing` flush with a per-dispatcher `problemsDispatched` flag (the dispatcher is created per review event by `createOutputPublisherResolverFromConfig`). Without the guard, the `allProblems` fallback re-publishes the full problem set with every summary entry, producing duplicate problem comments. Keep the `allProblems` fallback itself — it is required for PR-review channels configured only under `summary`, where `publishProblem` is never called.
48. **Model catalog bootstrap ordering and source tracking**: `bootstrapServerApp` must initialize `StoreDb` **before** resolving/enriching the `ModelSpec` and creating the gateway, and the store-init condition must cover admin auth **or** `llm.model_catalog` sqlite backend **or** reflection memory (not admin alone). The catalog service must call `ensureRefreshed()` (which also loads the bundled snapshot) before `enrichModelSpec`. The catalog backend (`ModelCatalogBackend.getEntry`/`getEntriesByModelId`) must return the stored `source` alongside each entry so enrichment can label `catalogSource` (`bundled` vs `remote`/`cache`) correctly — do not hardcode `"remote"`. When a config field gains a Zod default (e.g. `llm.model_catalog`), test fixtures that construct `AppConfig` by hand still break unless they include the field; read such fields defensively (`config.llm?.model_catalog?.enabled`) or update `makeConfig`.
49. **Agent sandbox timeout must kill the whole process tree, including `setsid` escapees**: The native and docker sandbox backends must terminate the entire descendant tree on timeout, not only the direct child. A `detached: true` spawn plus `process.kill(-pid)` (process-group signal) is **not sufficient on its own**: real agent binaries such as the Kilo CLI `setsid` their worker subprocesses into a brand-new session/process group (confirmed in production — orphaned workers had `pgid == sid != spawn group`), so a group signal never reaches them. They survive, get reparented to PID 1, hold the inherited stdio pipes, and cause a CPU-exhaustion death spiral (`durationMs` runs far past `agent.timeout_seconds`, e.g. 600s observed at 700s–1640s). `killProcessTree` in `packages/sandbox/src/process-tree.ts` must therefore, on Linux, walk `/proc` by PPID (which `setsid` does not change) and kill every descendant by PID deepest-first, in addition to the `kill(-pid)` group signal and `proc.kill(signal)`; Windows keeps `taskkill /T /F`. Keep the SIGTERM → SIGKILL cascade plus the force-resolve backstop that destroys stdio so the spawn promise can never hang. Verify with `packages/sandbox/test/native.test.ts` — both the inherited-stdio grandchild test and the `setsid`-worker regression test (the latter reproduces the production escape and is Linux-only). To recover a server already stuck in the spiral, restart the container so the runtime reaps all orphaned processes.
50. **The AICR container must run with `--init` so PID 1 reaps zombies**: `deploy/deploy.sh` starts the server with `podman run -d --init ...` (do not remove `--init`). Without it the Node server is PID 1 and never reaps descendants that escape the sandbox kill (e.g. `setsid` workers from pitfall #49) — they accumulate as state-`Z` zombies under PID 1 (observed 31 in production), and during the window before they exit they drive the death-spiral slowdown. `--init` runs `tini`/`catatonit` as PID 1 which reaps reparented zombies; it is the standard fix for "Node as PID 1" in containers. Both Podman and Docker support `--init`. If a deployment script or doc removes `--init`, reviews will eventually degrade into timeout failures again.
51. **Issue triage is Gitea/Forgejo-only and must be gated on the event provider family**: `resolveIssueTriageOptions` in `packages/server/src/bootstrap.ts` builds a single `GiteaApiClient` (which is API-compatible with both Gitea and Forgejo), so the trigger handler in `packages/server/src/index.ts` must run triage only when the issue event's `provider` is `"gitea"` or `"forgejo"`, and skip GitHub/GitLab/P4/SVN issue events entirely. Gate on the EVENT provider family, NOT on a tag derived from trigger `kind`: a Forgejo-kind trigger is served through the `/webhooks/gitea` route (`options.forgejo` is never populated, only `options.gitea`), so its events carry `provider: "gitea"`, and an equality check against the trigger kind would silently skip Forgejo triage. Otherwise GitHub/GitLab/P4 issue events are triaged through the Gitea client pointed at an internal/unsupported URL and fail with `issue_triage_failed` / `fetch failed` on every attempt. A GitHub triage path does not exist; do not wire non-Gitea workspaces into triage until a matching client is added.
52. **Git `fetchExtraContext` must fall back to `git show` for non-materialized related files**: `fetchScoped` only writes the CHANGED files (per-file `git show ${revision}:${path}`), so related-but-unchanged files the agent asks about (e.g. a header referenced by a changed `.cpp`) are NOT on disk. `fetchExtraContext` in `packages/vcs/src/git.ts` must therefore try `readFile` and, on ENOENT, fall back to `git -C <repo> show ${req.revision}:${path}`, persist the result into the workspace, then return it — mirroring the P4 adapter's `p4 print @revision` fallback. Without this, every related-file request ENOENTs, the orchestrator logs `ignored invalid fetch_more_context tool call`, the agent never gets the context it needs, and the review loops/skips/starves (production: `ENOENT .../source/.../atapp.h`). The orchestrator always passes `range.headRevision` as `req.revision`; use it. A `git show` that still fails means the path genuinely does not exist at that revision (or is a submodule gitlink) and the throw is the correct signal.
53. **Branch-create/delete push events must be skipped, not reviewed**: GitHub/Gitea send an all-zero SHA (40 or 64 zeros) for `before` on branch creation and for `after` on branch deletion. Neither has a reviewable commit range, and `git diff <sha>..<all-zeros>` fails with `fatal: Invalid revision range`, surfacing as `review_orchestration_failed` / `trigger processing failed`. `createPushReviewEvent` in `packages/server/src/webhook-common.ts` must return `null` (so the route answers `202 unsupported_event`) when `before` or `after` matches `/^0+$/u`. Direct commits to a branch (the common case the user wants reviewed) always carry two real SHAs and are unaffected.
54. **Parsers for external-command output must be validated against the real tool, not only synthetic fixtures**: `git blame --line-porcelain` emits a **4-field** group-start header `<sha> <orig> <final> <num_lines>` and a **3-field** coalesced header `<sha> <orig> <final>`; a regex anchored to exactly 3 fields (`…\s+(\d+)$`) silently drops every group-start line (and all size-1 groups), yet mock-fixture tests pass because hand-written fixtures used 3 fields throughout. `GIT_BLAME_HEADER_RE` in `packages/vcs/src/git.ts` must allow the optional 4th field (`(?:\s+\d+)?$`). When adding any parser for `git`/`svn`/`p4` output, run it against real command output (e.g. `git blame --line-porcelain -L 1,N <file>` piped through the built parser) before trusting unit tests.
55. **`daily_rollups` partition must be UTC day and every run/output mutation must trigger `recomputeDailyRollup`**: the dashboard time windows (today/thisWeek/thisMonth) are computed in **UTC** (`Date.UTC`/`getUTCDate` in `packages/server/src/observability-api.ts`), so `daily_rollups.date` must be the UTC `YYYY-MM-DD` of the run `started_at` (via `toUtcDateString`, i.e. `toISOString().slice(0,10)`), never local time — a local-day partition would silently misattribute runs near midnight and make rollup totals diverge from the real-time `getProjectStats`/`getOverviewStats` query path. `recomputeDailyRollup(store, projectId, date)` is the single idempotent recompute primitive (delete+insert in a `sqlite.transaction`, reading the raw `review_runs`/`code_metrics`/`llm_usage`/`output_events` tables); it is wired into `insertReviewRun` and `insertOutputEvents`. Any future code path that mutates those raw tables after the initial insert — most notably `updateRunStatus` if it is ever wired into production — MUST also call `recomputeDailyRollup` for the affected `projectId`+UTC date, or the rollup cache (a pre-aggregated view the dashboard may switch to) will go stale while real-time aggregation stays correct. The dashboard today still queries real-time aggregation as the source of truth, so a stale rollup is a silent correctness debt, not an immediate failure.
56. **Feishu cards MUST use the JSON 2.0 schema or inline code / code highlighting will not render**: the three Feishu card payloads in `packages/outputs/src/index.ts` (`createFeishuBotDispatcher` and the two `sendFeishuNotification` closures) must set `card.schema = "2.0"` and place markdown elements under `card.body.elements` (NOT directly under `card.elements`). The legacy 1.0 schema's supported-syntax table does NOT include inline code, so single-backtick spans like `` `GameplayTagsManager.h` `` render as literal backticks + text, and fenced code blocks get no language-based highlighting. Under 2.0, inline code (`code`), fenced code blocks with language parsing (```cpp …), headings, blockquotes, and tables all render natively, so `toFeishuMarkdown` in `packages/outputs/src/im-markdown.ts` must NOT downgrade headings→bold, blockquotes→italic, or tables→plain-text (those 1.0-era transforms actively harm 2.0 rendering); it should only run `fixAndValidateMarkdown` + blank-line collapse. The `@mention` append path must also write to `card.body.elements`, not `card.elements`.
57. **Problem lifecycle resolution must be guarded by reviewed file-scope and parse every emitted location shape**: a previously reported problem must not be resolved unless the current review re-analyzed its file. `reviewedFiles` must flow through `ReviewSummaryPublishOptions.reviewedFiles` → bootstrap `publishSummary` → the third `reconcileProblems` argument for GitHub and Gitea. Per-problem bodies embed `<!-- aicr:file=<path> -->`; consolidated headings emitted as either `file:line` or `file:start-end` must both round-trip through `parseConsolidatedBodyProblemInfo`, including `endLine`. Reconciliation is per fingerprint: reviewed-and-missing entries resolve, unreviewed entries remain in `open_problems`, and incomplete retained metadata skips the rewrite. Never replace this with whole-issue close-or-skip logic.
58. **Agent context compaction requires a known context window and must be injected per adapter**: `agent.context_compaction` (default `auto: true, prune: true`) flows from config through bootstrap (`contextCompaction` option) → orchestrator (`AgentBundleContext.compaction`) → `materializeRuntimeBundle` → each adapter's `materializeConfig(model, workingDir, options)`. Each adapter injects its native compaction format: Kilo writes `compaction.{auto,threshold_percent,prune}` into `kilo.json`; opencode writes `compaction.{auto,prune}` into `.opencode/config.json`; Zoo writes `autoCondenseContext` / `condenseContextPercentThreshold` into Zoo Code's current `.roo/settings.json` compatibility path; Claude Code auto-compacts by default (delegated); Copilot CLI has no surface (not_applicable). **Kilo only auto-compacts for models that declare a `contextWindow`** — if the model catalog is disabled and no `context_window` override is set, `buildKiloModelInfo` omits `contextWindow` and Kilo silently skips compaction, leading to `ContextOverflowError` on large PRs (production: P4 changelist requested 362661 tokens against a 262144 limit). Always enable `llm.model_catalog` or set `context_window` in overrides. When an overflow still occurs, the orchestrator's `runAgentReview` detects it (via `CONTEXT_OVERFLOW_RE` on both non-zero-exit stderr and the Kilo JSON-stream error event) and throws `AgentContextOverflowError` with the model limit, requested tokens, and actionable guidance — never let a raw `review_orchestration_failed` surface for an overflow. Bootstrap must also derive a default `compression` config from the model's `contextWindow` when `compression` is omitted (`toDefaultCompressionConfig`), and the orchestrator must compute `maxPromptTokens = floor(contextWindow × 0.6)` for the prompt-manager budget when `options.maxPromptTokens` is unset. Read `config.agent.context_compaction` defensively (`?.auto ?? true`) in bootstrap because hand-built test fixtures omit the field.
59. **The docs site is an isolated workspace package and must stay out of the runtime**: `docs/site` (`@aicr/docs-site`, Astro Starlight) is a documentation application, not runtime code. It must NOT appear in root `tsconfig.json` project references. `pnpm-workspace.yaml` must use the precise entry `docs/site`, never the `docs/*` glob (which would sweep plain Markdown topic docs into the workspace). Root `build` and `clean` scripts are filtered to `--filter "./packages/*"` so `pnpm -r` does not build the docs site; the docs site builds only via `pnpm docs:build` / `docs:check`. Those scripts run `docs/site/scripts/validate-public-content.mjs` before Astro so public pages cannot publish maintenance notes, `docs/ai/*`, `.agents/*`, or `AGENTS.md` references. `deploy/Dockerfile` does not COPY `docs/site` into the image; `pnpm install --frozen-lockfile` silently skips workspace importers whose directories are absent (verified empirically), so Astro/Starlight never enters the runtime image even though the lockfile lists them — do not add a docs/site COPY step. `.dockerignore` already excludes the whole `docs` tree. Starlight (≥0.33) requires `social` as an array of link items (not the legacy `{ github: "..." }` object), and frontmatter `template` only accepts `doc` or `splash`. Sidebar slugs omit the `index` segment: `troubleshooting/index.md` has slug `troubleshooting`. Content pages use `.md`, but the two landing pages (`en/index.mdx`, `zh-cn/index.mdx`) use `.mdx` so they can render Starlight components (hero frontmatter plus `Card`/`CardGrid`/`LinkCard`/`Steps`/`Aside`); MDX ships with Starlight with no extra integration (`@astrojs/mdx` is a transitive dep) and components never render in plain `.md` (they leak as literal `<CardGrid>` tags), so `validate-public-content.mjs` must scan both `.md` and `.mdx`. Only built-in Starlight icon names are valid for `Card`/hero `icon` (e.g. `rocket`, `server`, `puzzle`, `padlock`, `approve-check-circle`, `code-branch`, `analytics`, `comment`, `external`); invalid names like `key`, `terminal`, or `seti:docker` break the build. CI runs an independent `docs` job (`pnpm docs:build`) so a doc-build failure does not block the runtime `verify` job.
60. **GitHub App auth (M12) makes `ReviewOutputPublisherResolver` and `vcsFactory` async; all call sites must `await`**: `ReviewOutputPublisherResolver` in `packages/server/src/review-orchestrator.ts` returns `Promise<ReviewOutputPublisher | undefined>` and `vcsFactory` returns `Promise<DiffCapableVcsAdapter> | DiffCapableVcsAdapter`. The `GithubAppTokenService` (`packages/server/src/github-app-token.ts`) signs RS256 App JWTs with `node:crypto` (zero new deps), exchanges them for installation tokens, and caches/refreshes them (< 5 min lead). 61. **GitHub App installations must include every reviewed repository**: When switching a GitHub trigger to `app` auth, verify that the App is installed on the target account or organization and that every repository referenced by the trigger is selected in the installation. The GitHub API returns `404` for repos not in the selected set; AICR logs this as an installation-not-found error. Use the App's installation settings URL (`https://github.com/settings/installations/<id>` for users, `https://github.com/organizations/<org>/settings/installations/<id>` for orgs) to add missing repositories; the REST API endpoint to add repositories cannot be called with a GitHub App installation access token.
62. **Review problems must be deduplicated by fingerprint at every layer and problem-issue reconciliation must run once per review**: three independent defects combined to produce duplicate GitHub issues with doubled problem counts in production. (a) `collectCompletionOutputs` in `packages/server/src/review-orchestrator.ts` must NOT call `callAicrTools(completion.llmResult.content, ...)` when `completion.mcpState` already has final review output (`hasFinalReviewOutput`), nor when `completion.toolCallEvents` already produced `reviewOutputCount > 0`. The old code unconditionally re-parsed the agent's stream text after replaying MCP state, so the same `aicr.report_problem` records were collected twice (4 → 8). (b) `AicrOutputCollector.reportProblem` in `packages/mcp-output/src/index.ts` must fingerprint-dedup (by explicit `fingerprint` or `sha256(file:line:category:message)[:16]`) and skip duplicates; `clearReviewOutputs` must also clear the fingerprint set. (c) The `github_problem_issue` / `gitea_problem_issue` dispatchers' `reconcileProblems` in `packages/outputs/src/index.ts` must call `dedupProblemsByFingerprint` on the input `problems` array before building issue bodies or creating issues (both consolidated and per_problem modes). (d) The bootstrap publishers for `github_problem_issue` and `gitea_problem_issue` in `packages/server/src/bootstrap.ts` must guard `reconcileProblems` with a per-publisher `reconciled` flag (same pattern as pitfall #47's `problemsDispatched`): only the first `publishSummary` call triggers `reconcileProblems`; subsequent calls (from multiple summary entries in one review) return `[]`. Without this guard, a read-after-write gap in `listManagedOpenIssues` can cause the second call to create a duplicate consolidated issue. All four fixes are required; removing any one reintroduces the duplication under specific conditions.
63. **Consolidated cross-scope cleanup must reconcile per fingerprint, while `per_commit` stays isolated**: target-aware push scopes intentionally create a new issue for each `headSha`, so consolidated mode revisits older open scopes after the current issue is handled. Cross-scope mutation requires `resolvedAction !== "none"`, both commit markers, and `verifyCommitAtOrAfter(...) === true`; false or undefined, including compare failure, skips the old issue. Filter current findings to fingerprints already stored there, resolve covered-and-missing findings, retain uncovered ones, and preserve the old scope and historical summary. PATCH while open fingerprints remain; apply `resolved_action` only when the open set is empty. Current-scope-only fingerprints must never be copied into the old issue. Same-scope race duplicates are still resolved, empty reviews use the same per-fingerprint path, and `per_commit` never updates another commit scope.
